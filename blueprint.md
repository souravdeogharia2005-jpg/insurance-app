# AegisAI Architecture Blueprint

This is the master blueprint of the AegisAI platform. It details the complete folder structure, file relationships, API endpoints, and data flow of the application.

---

## 📂 Project Structure

The repository is organized into a clean monorepo structure with distinct `client` (Frontend) and `server` (Backend) directories.

### 💻 Client (React + Vite + Tailwind)
```text
client/
├── package.json             # Frontend dependencies (React, GSAP, Tailwind, html2pdf.js)
├── tailwind.config.js       # Global styling tokens, custom colors, animations
├── vercel.json              # Vercel deployment configuration (Routing rewrites)
└── src/
    ├── main.jsx             # React DOM entry point
    ├── App.jsx              # Main Router (BrowserRouter) wrapping all pages
    ├── index.css            # Global CSS (Imports Tailwind, sets @media print rules)
    │
    ├── assets/              # Static images, icons, and logo SVGs
    │   └── locales/         # i18n JSON files for multi-language support (EN, HI, FR, etc.)
    │
    ├── context/
    │   ├── AuthContext.jsx  # Manages user JWT tokens, login state, and persistent sessions
    │   └── AppContext.jsx   # Manages global state (Language, Theme, Currency formatting)
    │
    ├── components/
    │   ├── Navbar.jsx       # Global navigation, mobile hamburger menu, language toggle
    │   └── SplitText.jsx    # Advanced GSAP component for character-by-character animations
    │
    ├── pages/
    │   ├── HomePage.jsx     # Landing page with 3D elements and animated GSAP Hero
    │   ├── LoginPage.jsx    # Authentication flow (Login/Register toggles)
    │   ├── DashboardPage.jsx# Shows user's past proposals, pending quotes, and status
    │   ├── ScanPage.jsx     # The Core Underwriting UI: Camera, File Upload, AI Results, PDF generation
    │   └── AdminPage.jsx    # "Command Center" restricted route for overseeing/editing all proposals
    │
    └── utils/
        ├── api.js           # Centralized Axios functions for making requests to the backend
        └── emr.js           # The pure mathematical "Underwriting Engine" (BMI, Loading Factors)
```

### ⚙️ Server (Node.js + Express)
```text
server/
├── package.json             # Backend dependencies (Express, MySQL2, Google GenAI, JWT)
├── .env                     # Secrets: DB connection strings, JWT secret, Gemini API Keys
├── vision-key.json          # (Local Only) Google Cloud Service Account credentials
└── server.js                # Core API monolithic server file. Handles routing & DB connections.
```

## 🔀 Data Flow & API Routes

AegisAI uses a RESTful JSON API. All protected routes require a JWT Bearer token generated upon login.

### 1. Authentication (`server.js`)
* `POST /api/register`
  * **Input:** `{ username, password }`
  * **Action:** Hashes password with bcrypt, inserts into `users` table.
* `POST /api/login`
  * **Input:** `{ username, password }`
  * **Action:** Validates password, returns a signed JWT token for session persistence.

### 2. The AI OCR Pipeline
* `POST /api/vision-scan`
  * **Input:** `multipart/form-data` (Image File)
  * **Flow:** 
    1. Receives image buffer.
    2. Sends buffer to **Google Vision OCR API** using secure Base64 JWT credentials.
    3. Receives raw parsed text block.
    4. Sends raw text block + strict system prompt to **Google Gemini 1.5 Flash**.
    5. Gemini formats the text into a perfect JSON schema.
  * **Output:** `{ rawText: "...", structured: { age: 30, bmi: ... } }`

### 3. Proposal Management (`emr.js` + `server.js`)
* `POST /api/proposals`
  * **Input:** The final calculated EMR payload generated by `emr.js` on the client.
  * **Action:** Saves the newly underwritten policy into the MySQL database.
* `GET /api/proposals`
  * **Action:** Fetches all proposals belonging specifically to the logged-in `user_id`.

### 4. Admin Command Center
* `GET /api/admin/proposals`
  * **Action:** Bypasses `user_id` filtering to fetch the entire database of proposals.
* `PUT /api/admin/proposals/:id`
  * **Action:** Allows the Admin to override risk classes or approve/reject pending policies.
* `GET /api/admin/stats`
  * **Action:** Runs MySQL aggregation queries to calculate Monthly Revenue, Total Active Policies, and System Alerts.

## 📊 Database Schema (MySQL)

Hosted on minimal latency **Aiven Cloud MySQL**.

### Table: `users`
| Column | Type | Details |
| :--- | :--- | :--- |
| `id` | INT | Primary Key, Auto-increment |
| `username` | VARCHAR(255) | Unique index |
| `password` | VARCHAR(255) | Bcrypt Hashed String |
| `created_at` | TIMESTAMP | Auto-generated timestamp |

### Table: `proposals`
| Column | Type | Details |
| :--- | :--- | :--- |
| `id` | VARCHAR(36) | UUID v4 (Primary Key) |
| `user_id` | INT | Foreign Key mapped to `users.id` |
| `name` | VARCHAR(255) | Extracted applicant name |
| `age`, `height`, `weight`, `bmi` | FLOAT | Core biometrics |
| `emrScore` | INT | Total Calculated Risk Factor (0 = Best, >300 = Decline) |
| `riskClass` | VARCHAR(50) | Translates EMR to strings (e.g. "Class I", "Class IV") |
| `premium` | JSON | A serialized JSON object storing exact rupee amounts for Life/CIR/Accident policies |
| `status` | ENUM | `'pending'`, `'under_review'`, `'approved'`, `'rejected'` |
| `created_at` | TIMESTAMP | Auto-generated timestamp |

## 🎨 Design System Token Rules

If you need to extend the UI, adhere to these Tailwind conventions used throughout the app:
* **Backgrounds:** `bg-[#F8FAFC]` (Light Mode) / `bg-slate-950` (Dark Mode)
* **Accent Colors:** Primary actions use `bg-indigo-600` or `bg-slate-900`.
* **Success/Warnings:** `emerald-600` for good risk profiles, `amber-500` for warnings, `red-600` for fatal risks.
* **Containers:** Always use high-border-radius `rounded-2xl` or `rounded-[3rem]` paired with gentle `shadow-sm` and `border-slate-100` to maintain the "Modern Glass" aesthetic.
